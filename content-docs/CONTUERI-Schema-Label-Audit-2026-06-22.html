Schema Label and Description Audit

Date, June 22, 2026 · Scope, full audit across dailyPrompt, journeyDay, contentItem, traditionReflection · Method, grep against app/, components/, lib/sanity.ts to verify where each field actually renders.

Why this audit exists. Sheri noticed June 22 that the curatorNote field renders prominently on every P&P screen, but the Sanity Studio description ("Optional: why this was chosen for today. The human editorial voice.") reads as internal editorial metadata. The same drift pattern likely affects multiple fields. Goal of this audit, surface every label/description that no longer matches the actual app rendering, and propose corrections. No schema edits made in this audit, all changes deferred for Sheri's approval.
How to read the severity tags.
HIGH Label or description actively misleads about what the field is or where it renders. Affects editorial decisions.
MED Description is incomplete or slightly off, but not catastrophically wrong.
LOW Description is fine, may have a small phrasing polish.
UNUSED Field has zero app render hits. Either legacy, candidate for removal, or rendering missed by grep.

5. The deeper pattern

What this audit suggests beyond field-by-field fixes.

Pattern A. The three "hook" fields confusion. Three fields all carry a story-hook role on different surfaces, but the Studio labels and descriptions don't make their distinct roles obvious:

  • contentItem.artworkHook, current label "Hook (Encounter - 1st text)", renders ONLY on Journey Day Encounter
  • dailyPrompt.curatorNote, current label "Curator Note", renders prominently on P&P (the actual visible hook on every prompt)
  • journeyDay.encounterNote, current label "Encounter Note (Look Closer)", renders as the expandable "Look Closer" below the hook on Journey Day

The labels suggest one is THE hook and the others are something else. In practice all three carry hook-shaped content for different contexts. Better labels would name the surface explicitly: "Hook (Journey Encounter)" / "Hook (P&P, dark box)" / "Look Closer (Journey Encounter, expandable)."

Pattern B. Three "reflection questions" fields on three surfaces. Same issue, different fields:

  • contentItem.reflectionQuestions, renders on Visio Divina Pray step + Explore REFLECT expand
  • journeyDay.reflectionQuestions, renders on Journey Day Reflect step

Both have the same label "Reflection Questions" in Studio. An editor who sees both has no way to know which renders where without reading the description.

Pattern C. Two "audio" patterns for the same content. A piece can have audio in two distinct places:

  • contentItem.audioSource object (audioFile / audioUrl / externalUrl) for the PIECE's primary audio (music tracks, podcast episodes, lecture media)
  • contentItem.*Audio (artworkHookAudio, contextAudio, etc.) for narration of the WRITTEN text fields

The schema makes this distinction but Studio editors may not. A narration audio file uploaded to audioSource instead of the matching *Audio field would silently fail to render where the editor expected.

Pattern D. Surface-naming convention is missing. The audit suggests that every editorial field's label should include its rendering surface in parentheses, the way artworkHook already does ("Hook (Encounter - 1st text)") and curatorNote currently does NOT. The convention would be:

[Editorial role] ([Surface], [Position])

Examples: "Hook (Journey Day Encounter, dark box)" / "Curator Note (P&P, dark box)" / "Look Closer (Journey Day, expandable below hook)" / "Reflection Questions (Visio Divina Pray + Explore REFLECT)".

6. Recommended fix sequence

  1. HIGH-severity batch (5 fields, ~10 min): curatorNote, promptQuestion, artworkHook (description only; label was already renamed June 18), dayTitle, artworkHookAudio. Apply the surface-naming convention. Sanity Studio redeploy. Do this first.
  2. MEDIUM-severity batch (8 fields, ~15 min): encounterNote, lectio, context, description, reflectionQuestions (both), traditionalPrayer source, themes. Apply same convention.
  3. UNUSED fields audit (separate session, ~30 min): verify each of the 0-hit fields. Either confirm unused and hide / remove, or find the missing projection and document it.
  4. LOW-severity polish (optional): minor wording fixes only if doing a full sweep anyway.
  5. CLAUDE.md follow-up: add a "Field-to-Surface Map" section that lists which field renders where, so future Cowork or Claude Code sessions don't have to re-derive this from the schema descriptions. Could embed the table from this audit.
