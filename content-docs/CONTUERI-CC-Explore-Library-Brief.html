2. Explore Page — Task A: Physics Bubble Landing

What to replace

The current theme landing (the grid of aspect-[3/2] theme cards inside !selectedTheme) is replaced with the physics bubble UI. Everything else in the file — the header zone, map toggle, theme detail feed, ArtworkViewer, data fetching — stays untouched.

How it works (read the prototype)

Open contueri-explore-bubbles.html and interact with it. The bubbles are draggable, physics-driven circles on an espresso background. Each bubble represents a theme. Tapping (not dragging) a bubble transitions to the detail screen.

Key behaviors to preserve in the React port:

  • Bubbles use requestAnimationFrame for a physics simulation (velocity, damping, wall bounce, inter-bubble repulsion)
  • Drag vs. tap distinction — short-distance mouseup/touchend = tap (open detail), longer = drag (was panning)
  • The canvas/simulation dimensions should use the container's actual width, not a hardcoded 390 — this is one of the Task #12 fixes (see Section 4)
  • Bubbles render the theme name centered in the circle, colored with the theme's accent color
  • Bubble size is proportional to content volume (more content items = larger bubble)

React architecture guidance

Extract the bubble simulation into a custom hook (useThemeBubbles) or a separate component (ThemeBubbleCanvas). Keep the physics logic in a useRef/useEffect loop rather than React state to avoid re-render thrash on every frame. The prototype's vanilla JS is a clean reference — the simulation math ports directly.

Do not use innerHTML in the React port. The HTML prototype uses innerHTML to render bubble labels and card content — this is flagged as a security issue for App Store review. In React, render bubble text and card content with JSX elements only.

3. Explore Page — Task B: Option E Editorial Detail Header

When a user taps a bubble, the current code renders the theme title as a plain h1 inside the main header bar alongside the back button. This is replaced with a two-zone editorial header — the locked Option E design.

Two-zone structure

Zone 1 — Slim nav chrome (always visible, ~46px tall):

  • Back chevron (left) → returns to bubble landing
  • CONTUERI wordmark centered (Montserrat 600, 12.5px, letter-spacing 0.24em, uppercase, rgba(26,26,26,0.58))
  • Map icon (right) → existing map toggle behavior
  • Parchment background, 0.5px bottom border at rgba(26,26,26,0.07)

Zone 2 — Editorial header (below Zone 1, text-centered):

  • Theme title — Montserrat 600, 26px, letter-spacing: 0.14em, text-transform: uppercase, color = theme's accent color (from theme.color), not near-black
  • No rule — there is no decorative line under the title in Option E
  • Theme question — Open Sans, 14px, italic, color: #6e8e7e, line-height 1.55, max-width 270px centered, margin-top: 20px (this spacing compensates for the absent rule)
  • Padding: 24px top, 32px sides, 22px bottom
  • Parchment background, 0.5px bottom border at rgba(26,26,26,0.09)

Editorial gallery cards (content feed below the header)

The existing 16:9 image cards in the theme detail feed are replaced with 1:1 square editorial cards. Each card:

  • Image: aspect-ratio: 1 / 1, full-bleed, no rounded corners
  • Type label: Montserrat 600, 8px, uppercase, letter-spacing 0.18em, sage color (#7a9a8a)
  • Item title: Cormorant Garamond, 24px, weight 400, color: #1a1a1a, line-height 1.2
  • Attribution: 11px, sage-muted, below title
  • Excerpt/description: Open Sans, 13px, italic, color: #3d3530, line-height 1.65, 3-line clamp
  • Card padding: 14px 18px 20px
  • No prop-rule — no decorative line between image and text
The existing ArtworkViewer component is still used when a user taps a content card inside the detail feed — that behavior is unchanged. The editorial card is just a new presentation layer for the list item.

4. Explore Page — Task C: Production-Readiness Fixes (Bundle with Tasks A & B)

These were identified in a prior CC Opus audit (Task #12 in the backlog). Since Tasks A and B already touch the explore page, fix these in the same PR rather than a separate pass.

Priority Issue Fix
Critical Reduced-motion fallback broken Check prefers-reduced-motion media query. When true, skip the physics animation — render bubbles as a static grid instead.
Critical innerHTML used for bubble labels and card content Already solved by the React port — render with JSX, never innerHTML.
High requestAnimationFrame loop runs forever even when off-screen Cancel the rAF loop (cancelAnimationFrame) when the component unmounts or when the detail screen is visible. Resume when returning to the bubble view.
High No ARIA / keyboard navigation Each bubble should be a <button> with aria-label="{theme.title} — {theme.question}". Tab order should cycle through themes. Enter/Space opens detail.
High iOS ghost-click on bubble tap Add touch-action: manipulation to bubble elements. Use touchend with e.preventDefault() to suppress the 300ms delayed click.
High touchcancel not handled Add a touchcancel listener that clears the active drag state — same cleanup as touchend.
Medium Canvas width hardcoded to 390 Use containerRef.current.getBoundingClientRect().width on mount (and on resize via ResizeObserver) to set simulation width dynamically.
Medium Viewport meta needs viewport-fit=cover Update app/layout.tsx viewport meta: content="width=device-width, initial-scale=1, viewport-fit=cover". Ensures safe-area insets work on iPhone with notch/Dynamic Island.

7. Build Verification

After all changes:

  1. Run npm run build from repo root. If the SWC binary error appears (Cowork VM ARM64 limitation), fall back to npx tsc --noEmit — but note that tsc alone does not catch all Next.js build failures. Prefer the full build wherever possible.
  2. Visually test the explore page: bubbles animate, drag works, tap opens detail, back returns to bubbles, map toggle works, reduced-motion renders a static fallback.
  3. Confirm the Option E header: "LIGHT" (or any theme title) appears uppercase in the theme accent color with no rule below, and the italic question has breathing room beneath it.
  4. Commit after each task (A, B/C together, library) so there are clean rollback points.
Do not claim any task is shipped on tsc --noEmit alone. It misses Suspense wrapping, prerender errors, and dynamic-route mismatches. Always prefer npm run build.
