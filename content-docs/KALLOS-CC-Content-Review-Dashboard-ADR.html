A sixth cross-cutting principle emerged: presets are parameterized, not static. Each preset pre-selects its column set and exposes one to three parameters for the user (which journey, which field, which day). This collapses "five dashboards" into one grid with five saved configurations.

Cross-cutting principle — parameterized presets

What might feel like five separate dashboards are five preset configurations on a single grid. Each preset decides the opinionated starting point (which columns to show, which identifiers are locked on the left). The user picks one parameter from a dropdown to define the slice, and from there adds or removes columns as needed.

PresetDefault columnsParameter
Record viewAll fields, sequential orderWhich record
Field comparison2 identifiers + 1 focus fieldWhich field, which record type
Journey arc2 identifiers + structural fieldsWhich journey
Parallel entry point2 identifiers + tone fieldsWhich day number
Content pairing2 identifiers + chosen field pairWhich pair of fields

Identifiers are always pinned on the left and always present in every preset's default columns. For journeyDay: Journey Title plus Day label (formatted as "Day N: {dayTitle}", e.g. "Day 5: They saw dimly"). For dailyPrompt: Date plus Artwork Title. The first identifier column is a link that navigates to record view.

Default sort groups rows by container. journeyDay rows sort by Journey Title ascending, then Day Number ascending, so all 7 Myth days appear in sequence, then all 8 Bosch days, and so on. dailyPrompt rows sort by Date descending. This keeps the grid readable as content grows past dozens of records. User-selected sort overrides the default.

A sixth button labeled Custom sits alongside the five presets. All columns off by default, all filters available, no preselection. For the unusual session that does not match a preset.

The five decisions

Q1 Where to live
Decision
Build inside the existing Next.js app, under a new route (e.g. /dashboard/review). Do not use the Sanity App SDK for Phase 1.

Why. The App SDK is less than a year old. New platforms on new APIs are moving targets, and this is a tool you use during real content sessions. The existing dashboard is known-good infrastructure: GROQ queries in lib/sanity.ts, server components, revalidate: 60, the DASHBOARD_ENABLED gate, all proven. The "alongside Studio" intuition is overstated. Studio itself is a split layout; cramming a cross-record review tool into a side panel inside it is probably worse than a dedicated full-viewport view on its own tab.

Migration path. If the App SDK matures and the tool's design stabilizes, the GROQ queries transfer with no rework. Moving later is low-cost. Committing to App SDK now is high-cost if the API shifts.

No conflict with Presentation plugin. Both can coexist in Studio; the review tool simply does not live inside Studio at all in Phase 1.

Q2 Cross-document-type rows
Decision
Single GROQ query that unions two typed projections. TypeScript row type is a discriminated union keyed on _type.

Query shape. One round-trip, two projections concatenated server-side:

[...*[_type == "journeyDay" && filter]{...journeyDayFields}, ...*[_type == "dailyPrompt" && filter]{...dailyPromptFields}]

Type shape. GridRow = JourneyDayRow | DailyPromptRow. Each variant carries only its own fields. Renderers branch on _type. Do not use a merged type with all fields optional; it silently allows rendering promptQuestion on a journeyDay row.

Name-collision policy. dailyPrompt.curatorNote, contentItem.artworkHook, and journeyDay.encounterNote are three distinct jobs (per R1 Option C, already documented in CLAUDE.md). In the grid they appear as three separate columns with labels that state which type the field belongs to. Never merged.

Q3 Long content handling and the detail surface
Decision
Inline cell expansion in the grid (single click to expand, double click to revert, drag-to-resize column headers). Record view is its own URL at /dashboard/review/record/[id], reached by clicking the first identifier column.

Why inline over slide-over or modal. Field comparison requires reading 5 to 10 cells of the same field at the same time. A slide-over or modal blocks the grid and forces one-at-a-time reading, which defeats the comparison workflow. Sheri's reference dashboard (quarterly reporting with narrative Story column) handles this correctly with the taller-cell plus click-to-expand model. Inline expansion is the right pattern for all four comparison-oriented presets.

Why record view lives at its own URL. Reading one record as a full sequential experience is a different layout, not a different grid state. A dedicated page lets the fields render top-to-bottom in user-sequence order (image, openText, artworkHook, context, lectio, auditio, reflectionQuestions, connectThread, goDeeper), full viewport, bookmarkable, shareable, print-friendly. It also allows the operational /dashboard to link into record view for audit-flagged records, which costs nothing and adds a direct path from "what is missing" to "read the whole thing."

Ref traversal. Pre-join referenced fields in the grid query via projection. Never a second fetch on click. For journeyDay rows, project encounterContent->artworkHook, encounterContent->context, and encounterContent->image.asset->url as part of the initial payload. Detail panel reads from the row object the grid already has.

Cell expansion state. Serialized in URL as &expanded=rowId:colKey,rowId:colKey. Column widths serialize similarly if persistence is wanted. Aligns with Q4 URL-first commitment.

Q4 Preset persistence and URL state
Decision
URL params are the source of truth for preset, parameter, column toggles, active filters, pinned rows, and cell expansion state. Presets are hardcoded in a small registry. No user-saved presets in Phase 1.

Why URL over localStorage. Shareable ("send me the view you are looking at"), bookmarkable, survives incognito, works across devices, browser back and forward navigate between views for free. localStorage breaks all of these and adds a persistence layer that complicates the read-only promise.

Preset registry. A small config file lists the five presets with their default columns and parameter schemas. Adding a preset is a code change. Saving custom views is a bookmark.

URL shape example. ?preset=arc&journey=bosch&cols=openText,connectThread,auditio. Preset change = navigate to new URL. Parameter change = URL update, client-side refilter. Column toggle = additive override on top of preset defaults, serialized into URL.

Technical debt avoided. The only real Phase 1 debt risk is storing filter or column state in useState instead of URL. Commit to URL from day one. Retrofitting URL sync later is painful; building it in from the start costs nothing.

Q5 Data fetching strategy
Decision
Follow the existing /dashboard pattern. Named GROQ function in lib/sanity.ts, server component fetches all rows eagerly with all projections, passes to a client component that handles filter, sort, column toggle, and cell expansion entirely in memory. revalidate: 60 unchanged.

Why eager fetch at current scale. 52 rows today, projected 250 by year-end. Full payload per row with projected ref fields is roughly 1 to 2 MB worst case. One page load, every interaction after that is instant client-side JavaScript over already-loaded data. No pagination, no virtualization.

Why not live updates. Editorial review is not operational monitoring. If Sheri just published an edit, she refreshes the page. Live sync would add websockets or polling complexity for no user-visible benefit at this scale.

Scale pivot point. If rows cross roughly 1,000 and pages start feeling slow, switch to row virtualization (react-virtuoso or equivalent) before paginating. Pagination breaks cross-record comparison by definition. This is 12+ months out at current velocity and not a Phase 1 concern.

Future-feature foreclosure check

Phase 1 is read-only. The brief asked to flag any decision that would paint future features into a corner. None of the above does.

  • Editorial tagging: when it lands, add an editorialPrinciples ref array or inline array to journeyDay and dailyPrompt schemas. Do not build a separate annotation store. The Sanity fields become the source of truth; the grid reads them as any other column. URL-param preset architecture accommodates a "filter by principle" parameter with zero rework.
  • Accept, reject, or modify for similarity detection: write functionality, a different architecture (Next.js server actions plus Sanity mutations with a write token). Phase 1 read-only does not foreclose it. Treat it as a new subsystem when it lands.
  • CSV export: trivial to add. A /dashboard/review/export route that re-runs the grid query with the same URL parameters and returns CSV. Recommend shipping this in Phase 1 release (small addition, high leverage as an escape hatch for anything the grid does not cover).
  • TTS "really listen" view: entirely separate surface, time-based. No data-model overlap. Not allowed to influence grid decisions.
  • Comments and annotations: deferred per the brief. No Phase 1 impact.

Phase 1 scope

In Phase 1

  • New route: /dashboard/review (grid) and /dashboard/review/record/[id] (record view).
  • Single GROQ query unioning journeyDay and dailyPrompt, with ref projections for contentItem fields.
  • Five named presets plus Custom, each with parameter dropdown.
  • Column toggle menu, drag-to-resize column widths, inline cell expand / revert, row pin, filter controls.
  • All state serialized in URL parameters.
  • Record view page: full viewport, fields in user-sequence order, "Back to grid" preserves URL state.
  • Link into record view from relevant /dashboard audit tables (small "Review" link per row).
  • CSV export route that reuses the same query plus URL parameters.
  • Voice-rule scanner (QA layer): text cells are scanned against a config list of banned patterns from the KALLOS voice rules. Cells with matches get a small indicator badge; hovering reveals which pattern tripped it. Quick-filter chip at the top of the grid labeled "Voice flags" narrows the grid to just offending rows. Banned patterns for Phase 1 (all from CLAUDE.md editorial rules):
    • Em dash (—) — telltale AI-generated text, banned everywhere in KALLOS.
    • Banned voice words: "journey" (as noun), "sacred", "profound", "transformative", "spiritual practice", "invitation to".
    • "Shaft" when describing light — use "ray" or "stream" instead.
    Architecture: single config array of { pattern, label, severity, suggestion } objects. Each cell runs all patterns on render; matches aggregate into one badge per cell. Adding a new banned pattern is a one-line config change (no code edits elsewhere). Supports future expansion to generic AI-tell detectors ("delve", "tapestry", "moreover", "at the heart of", etc.) without structural rework.
  • Gated by existing DASHBOARD_ENABLED env flag. noindex, nofollow.

Out of Phase 1

  • No tagging, no write operations, no comments, no annotations.
  • No user-saved custom presets (bookmark the URL instead).
  • No similarity detection, no automated candidate surfacing.
  • No TTS or audio playback view.
  • No live updates or websocket sync.
  • No Sanity App SDK port.

Constraints carried forward

  • Sheri does not write code. The tool runs entirely in the browser.
  • Phase 1 is read-only. No editing, no saving, no writes.
  • Existing operational /dashboard continues working. The review tool is additive.
  • No em dashes in code comments or content, anywhere.
  • Build verification: npm run build; fall back to npx tsc --noEmit on Cowork VM. Commit after every significant step.
  • Must be fast enough to use alongside Sanity Studio during content sessions.
