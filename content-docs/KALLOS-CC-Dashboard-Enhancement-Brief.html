Task 3 Dashboard — filter and sort Must do

The dashboard is currently a server component. Server components cannot handle interactive state. Filter and sort require client-side interactivity. The solution is to extract the sections that need filtering into client component wrappers that receive already-fetched data as props — the same pattern used elsewhere in the app (e.g. app/prompt/PromptClient.tsx).

Architecture

app/dashboard/page.tsx remains a server component and fetches all data. It passes data as props to new client components:

  • app/dashboard/TRTableClient.tsx — handles the tradition reflection table with filters and sort
  • app/dashboard/AudioTableClient.tsx — handles the audio status table with filters
  • app/dashboard/ContentItemsClient.tsx — handles content item table with filters

Filters per section

TR table: Filter by journey / by type (church-father, philosopher, etc.) / by era / by theme / has audio or missing audio / KEEP or REVIEW (artworkHook audit status)

Audio table: Filter by journey / show missing only (most useful — "show me everything that still needs recording")

Content items: Filter by content type / by theme / missing artworkHook / missing audio

Sort

All table columns should be sortable by clicking the column header. Clicking again reverses sort direction. Show a sort indicator (arrow) on the active sort column.

Implementation notes

Use React useState for active filter values and sort state. No URL params needed for now — in-memory state is sufficient. Filters reset on page refresh, which is acceptable for a dashboard tool.

Task 5 Sanity Presentation plugin — live preview while editing Stretch
Stretch goal
Only attempt Task 5 if Tasks 1-4 are complete and committed. If time is short, stop at Task 4 and carry Task 5 to the next session.

Currently, Sanity Studio has click-to-open preview URLs (configured April 13, 2026 via resolvePreviewUrl() in sanity/sanity.config.ts). This task upgrades to an inline real-time preview pane — while editing a journey day or daily prompt on the left, the rendered app view updates live on the right as you type.

What this enables

Sheri sees exactly how the content item, journey day, or P&P prompt looks in the app in real-time while editing — solving the problem that "content input should mirror the UX screens for the content type being created." No more guessing whether a curator note is too long or whether the context field renders correctly.

Implementation

  1. Install: cd sanity && npm install @sanity/presentation
  2. Add the presentationTool() plugin to sanity/sanity.config.ts, configured with the existing resolvePreviewUrl() helper already in place
  3. Add a SANITY_PREVIEW_SECRET to .env.local and Vercel environment variables
  4. Add a Next.js draft mode route handler at app/api/draft/route.ts that validates the preview secret and enables draft mode
  5. Update the Sanity client in lib/sanity.ts to support a previewDrafts perspective when draft mode is active
  6. Test: open a journey day in Sanity Studio, verify the Presentation panel shows the live rendered app view updating as fields are edited

Scope

Configure live preview for: journeyDay, dailyPrompt, contentItem. These are the three document types Sheri edits most frequently. The existing resolvePreviewUrl() helper already maps these to the correct app routes — extend it to support the Presentation plugin's requirements.
