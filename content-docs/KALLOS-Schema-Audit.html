KALLOS SCHEMA AUDIT

Task 1 output — field inventory, duplicate detection, GROQ trace, recommendations
April 23, 2026 — Read-and-report. No schema or code changes made.
What this document is: A full map of KALLOS's Sanity schema architecture, the fields that are entered in the Studio, the GROQ queries that pull them, and what the app actually renders. Where the three diverge, there is a bug, a dead field, or an editorial trap. This audit documents all three.

What it isn't: A plan to fix anything. All recommendations are labeled and prioritized, but nothing in the schema or code was modified during this session. Approve recommendations in a separate working session before any change is implemented.

Executive Summary

The schema is in mid-migration state. The new KALLOS schemas (theme, contentItem, journey, dailyPrompt) are live and carry all production content. The old schemas (episode, the unregistered artwork.ts file) linger with migration helpers still in place. contentItem itself still carries seven "legacy" fields kept for backward compat. This is expected during migration, but the debt is now structural and worth addressing before companion journeys expand the schema surface.

The five most important findings

  1. journeyDay is not a standalone schema. It is an inline object type inside journey.days[]. You cannot query *[_type == "journeyDay"]. Every day is addressable only via its parent journey. CLAUDE.md's Schema Status table implies otherwise.
  2. The brief's premise is not literally true. contentItem and journeyDay do not share field names for curatorialNote, context, or encounterNote. The overlap is conceptual (parallel slots under different names), not identical. The real editorial trap is that Journey Day stacks three overlapping prose fields on the same screen (content.curatorNote + content.context + day.encounterNote), with no schema guard against duplication.
  3. Two live rendering bugs. (a) contentItem.curatorNote is fetched but never rendered on the Pause & Ponder page — the app only shows dailyPrompt.curatorNote. So if an editor puts the hook on the content item instead of the prompt, it vanishes. (b) traditionalPrayerSource is rendered by the Visio Divina Pray step but never projected by the GROQ query — it will always be undefined.
  4. Near-duplicate field names are a typo waiting to happen. reflectionQuestions (contentItem) and reflectQuestions (journeyDay) live one letter apart. Both are in live use.
  5. Dead code is accumulating. sanity/schemaTypes/artwork.ts is orphaned (not in index.ts). episode is registered but no GROQ query reads it. Seven legacy fields on contentItem are hidden but still on disk. Zero risk today; increasing confusion as the schema grows.

1d. Dead Fields

Fields defined in the schema that are never pulled by any GROQ query in lib/sanity.ts. Grouped by severity.

Active bugs — rendered by UI but not fetched Priority 1

  • contentItem.traditionalPrayerSource — rendered at app/pray/[artworkId]/page.tsx:486. Missing from getArtworkById GROQ projection. The attribution will never appear.

Fields defined but not projected or rendered Priority 2

  • contentItem.order — hidden, no query, no render. Candidate for removal.
  • contentItem.locationType — legacy, hidden, no query, no render. Candidate for removal once legacy data migrated.
  • contentItem.historicalSummary — legacy, hidden, only used by getArtworkById. Candidate for removal once Visio Divina doesn't need the fallback.
  • traditionReflection.theme (legacy string) — still pulled by getTraditionReflections but the field is hidden in Studio and replaced by themes. Remove from query or remove field.

Fields projected inconsistently Priority 2

  • traditionReflection.themes — pulled in getTraditionReflections but NOT in the goDeeper projection of getJourney or the traditionReflections projection of getArtworkById. Any dashboard or UI that wants to group TRs by theme from those contexts will get empty arrays.
  • contentItem.traditionReflections — pulled only in getArtworkById. Not in CONTENT_ITEM_FIELDS. So Journey content cannot expose content-level Go Deeper references; only Visio Divina can.
  • contentItem.era, tradition, discipline, principle, series, craftTradition, pointsToward, creationTheology, beautyConnection, durationMinutes — all projected by CONTENT_ITEM_FIELDS but I did not verify their render paths in the UI audit. They may be displayed only on certain content types; worth spot-checking during content inventory.

Schemas and files with no usage Priority 2

  • sanity/schemaTypes/artwork.ts — file exists, not registered in index.ts. Pure dead code. Recommend: delete.
  • episode schema — registered in index.ts but no GROQ query reads it. Studio still shows existing documents; the app ignores them. Recommend: either migrate episode data to contentItem records or unregister the schema.

1e. Recommendations

Grouped by priority. Nothing in this section has been implemented. Each recommendation includes what to change, why, and what breaks if it is left alone.

Must fix before companion journeys are built (Priority 1)

R1Decide the canonical location for curatorNote and fix the rendering asymmetry
What

Either (a) delete contentItem.curatorNote and contentItem.curatorNoteAudio and make dailyPrompt.curatorNote + per-day curator voice the only home for today's editorial framing, OR (b) render content.curatorNote on P&P when prompt.curatorNote is absent (fallback chain), so the field has a predictable meaning regardless of where it is entered.

Why

Editors don't know that contentItem.curatorNote is silently ignored on P&P. The CLAUDE.md rule "Curator Note = the hook, lives here and ONLY here" implies one slot; the schema has two. Companion journeys will multiply this drift.

Risk if untouched

Content authored on the content item appears on Journey but not on the same item's P&P day. The Lectio standard depends on editors being able to predict where the hook shows up. They can't, today.

R2Fix traditionalPrayerSource — add to GROQ projection
What

Add traditionalPrayerSource to the getArtworkById projection in lib/sanity.ts. Three-line fix. Sheri should still approve the change first.

Why

The UI renders it. The schema defines it. The query does not fetch it. Silent bug.

Risk if untouched

Traditional prayer attributions never appear on the Visio Divina Pray step, even when entered in Sanity.

R3Unify lectio shape across journeyDay and dailyPrompt
What

Pick one subfield vocabulary and apply it to both. Recommended: dailyPrompt's (philosophyText, philosophyAttribution, text, attribution) since CLAUDE.md documents it as the Lectio pairing structure. Add connectionNote to dailyPrompt.lectio if the editorial team wants that field there too. Migrate existing journey data. Journey content doc authors will still need a pairing doc explaining the schema.

Why

Companion journeys will inherit the journey schema. If Lectio is going to be a first-class editorial concept across P&P, Journey, and companion content, its schema should be the same shape in every place. A single GROQ function can then report on all Lectio pairings.

Risk if untouched

Every dashboard, export, and analysis script must special-case both shapes. The schema documents drift further each time a field is added on one side.

R4Unify auditio shape across journeyDay and dailyPrompt, and flatten the nested audioUrl
What

Same subfield names on both schemas. Recommended: flat shape { title, composer, licensingNote, audioFile, audioUrl, externalUrl, verbaOriginal } — taking the clearer parts from each. Eliminate the confusing dailyPrompt.auditio.audioUrl object-with-an-audioUrl-string-inside.

Why

TTS audits, content inventories, and any future audio-only dashboards need one query path. The nested audioUrl object is a bug magnet.

Risk if untouched

Same as R3, plus increased risk of query typos during the companion journey build.

R5Decide whether journeyDay should become a standalone document schema
What

Currently journeyDay is an inline object. Promote it to a document schema with its own _id, add a journey reference field, and change journey.days to an array of references. This is a schema migration with real data rewrite cost — not a quick fix.

Why

As a document, days would (a) be queryable independently (*[_type == "journeyDay"]), (b) have stable IDs for deep-linking and analytics, (c) allow content items to reference back to days they appear on, (d) support cross-journey reuse of a day, (e) make Sanity Studio navigation and reuse much stronger. Dashboard Section 1 in Task 3 would be cleaner and faster.

Against

Migration costs time and has some risk. The current inline design is simpler to author (you see the whole journey in one Studio document). Some teams prefer the nested structure. No rush if companion journeys don't need cross-journey day references.

Recommendation

Decide in the post-audit working session. If companion journeys will reuse days or need per-day analytics, promote. If companion journeys are fully self-contained, leave as is. Document the decision in CLAUDE.md either way.

Should fix soon (Priority 2)

R6Rename reflectQuestions to reflectionQuestions on journeyDay
What

Align the field names. Requires schema change + data migration + component update + seed script updates.

Why

The one-letter-apart split is a pure typo hazard. No editorial or UX justification for the difference.

Risk if untouched

Every new content doc, seed script, and dashboard is a chance to ship a bug.

R7Align era vocabulary across contentItem and traditionReflection
What

Pick one list. Recommended superset: ancient, patristic/fathers (unified under one label), medieval, renaissance, modern, contemporary. Apply to both schemas. Add "ancient" to traditionReflection.era (CLAUDE.md manual task 37 already notes this).

Why

The Go Deeper editorial rule says every journey day should include "one ancient / patristic, one medieval, one modern" voice. Unified vocab makes this reportable; currently it is not.

Risk if untouched

Dashboard era breakdowns will have gaps. Content diversity rules can't be enforced automatically.

R8Change traditionReflection.authorType to multi-select
What

Schema currently uses layout: 'radio' (single-select). Change to array-of-string with checkbox layout, preserving the 7 values. Migrate existing records to array-of-one.

Why

Aquinas is a saint AND a Doctor of the Church. Augustine is a Church Father AND a Doctor. CLAUDE.md Schema Status table flags this. Multi-select supports the reality.

Risk if untouched

Content authors have to pick one label and lose the other. Dashboards under-count roles.

R9Project themes consistently across all tradition reflection queries
What

Add "themes": themes[]->{...} to the goDeeper projection in getJourney and to the traditionReflections projection in getArtworkById. Matches the shape used by getTraditionReflections.

Why

The theme badge/filter mechanism works in one list context but not in any nested context. Dashboards and any "thinker tagged with theme X" view need this.

R10Decide fate of episode schema
What

Either (a) migrate existing episode documents to contentItem with contentType = "watch-listen" and unregister the schema, or (b) keep the schema registered only as a data lifeboat but add a comment saying no app surface reads it.

Why

Dead schema with live documents is confusing in the Studio sidebar. Either finish the migration or formally accept the data is archival.

R11Delete sanity/schemaTypes/artwork.ts
What

Remove the file. Keep the _type in ["contentItem", "artwork"] fallback in GROQ for one more release cycle in case legacy artwork documents still exist, then remove that fallback once verified empty.

Why

Pure dead code. Confusing.

R12Document the parallel-slots trap in CLAUDE.md
What

Add a section to CLAUDE.md (or CONTENT-RULES.md) that spells out, field by field, where each piece of prose lives: "Hook = dailyPrompt.curatorNote on P&P days, contentItem.curatorNote on Journey days. Historical context = contentItem.context. Look-closer symbolism = journeyDay.encounterNote." Include what NOT to repeat.

Why

Until the schema is unified, the rule set is the only thing preventing editorial duplication. It should be written down.

Can wait (Priority 3)

R13Remove legacy contentItem fields after data migration
What

Audit every published document for non-empty locationType, historicalSummary, order, quote.attribution (legacy subfield), musicUrl, audioFile (top-level legacy). Migrate content forward. Remove fields from schema.

Why

Reduces schema surface area and Studio clutter. Low risk; no immediate harm if left.

R14Remove legacy traditionReflection.theme string field
What

Remove the hidden legacy field and drop it from getTraditionReflections.

Why

Replaced by themes array. Still projected in a query, still on disk.

R15Refactor contentItem into type-specific subdocuments (long term)
What

The current single contentItem schema holds fields for all 8 content types with hidden conditionals. Consider splitting into per-type schemas (sacredArtItem, thinkerItem, musicItem, …) with a shared base.

Why

Reduces field count per document (~40 now), removes the need for conditional-required validations, and lets each type have its own Studio UI. Photography (pending manual task 36) could be added as a ninth type cleanly.

Against

Large refactor. Breaks existing queries. Explore's unified content feed would need type-fanout. Not urgent.

Recommendation

Defer. Revisit after companion journeys ship or if the unified model starts producing bugs.
