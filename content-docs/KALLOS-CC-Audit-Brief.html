Task 1 Schema Architecture Audit Priority 1

Map the full schema architecture and identify structural problems. This is the prerequisite for everything else.

1a. Field inventory across all schemas

For every schema in sanity/schemaTypes/, list every field: field name, field type, whether it is required, and whether it appears in any other schema under the same name. Produce this as a table.

1b. Cross-schema duplicate detection

The known problem: contentItem and journeyDay share field names (curatorialNote, context, encounterNote, etc.). Some content was entered in both places by mistake because the fields have the same name and it was initially unclear which one the app displays. Identify every field that exists in both schemas and flag it.

1c. GROQ query trace

For each shared/duplicate field found in 1b: trace which GROQ query in the app actually pulls that field and renders it. Check lib/sanity.ts, app/journeys/, app/prompt/, and any other data-fetching files. For each duplicate field, answer: is the app using the contentItem version, the journeyDay version, or both? Flag any field where both versions might be displayed simultaneously.

1d. Dead fields

Identify any fields that exist in the schema but are never referenced in any GROQ query. These are either unused or they should be added to queries.

1e. Schema health summary and recommendations

Produce a plain-language summary of current state. Then for each issue found, provide a specific recommendation: what should change, why, and what the risk is if it stays as-is. Prioritize: what must be fixed before companion journeys are built, what should be fixed soon, and what can wait. Do not implement any recommendation — document it only. Sheri will review and approve before any changes are made.

Output
Save as content-docs/KALLOS-Schema-Audit.html — a structured report with tables. Use the KALLOS HTML style (dark espresso header, gold accents, field tables). No code changes.

Query Sanity for all entered content and produce a complete inventory. Use the Sanity API with the project ID above. All queries use GROQ.

Reporting depth for all sections

Apply these rules to every field on every document type throughout Tasks 2a-2f. Different field types have different definitions of "missing" — do not default to text-only checks.

  • Text fields: Populated = yes / no / partial (partial = content present but under 20 characters for a field that should be a full sentence). Preview = first 80 characters. Do not return full content.
  • Single-value selection fields (e.g. contentType, authorType, era, medium): Populated = yes if an option was selected, no if the field was left unselected. "No selection made" is a missing field — treat it the same as blank text.
  • Array fields (e.g. themes, tags): Populated = yes if the array has at least one entry, no if the array is empty or absent. Report entry count when present. An empty array is a missing field.
  • Image and audio upload fields: Populated = yes if an asset is attached, no if empty. Report asset filename when present.
  • Reference fields (e.g. journey association, day association, linked content item): Populated = yes if linked, no/unlinked if the reference is absent. An unlinked record that should belong to a parent is an orphan — flag it separately in the consolidated red list.

These rules apply universally. Section 2d has additional analysis on top of these rules, but the same completeness definitions apply to journey days, content items, daily prompts, and every other document type.

2a. Journey and journey day inventory

For each journey: title, slug, totalDays field value, number of journey day records actually entered, completion percentage. For each journey day within each journey: title, and the populated/preview status of every field found in the journeyDay schema (Task 1 output). Group days under their parent journey.

2b. Content item inventory

For each content item: the populated/preview status of every field found in the contentItem schema (Task 1 output). Group by contentType. Show count per type at the top of each group.

2c. Daily prompt (P&P) inventory

For each daily prompt: date, title, and the populated/preview status of every field found in the dailyPrompt schema (Task 1 output). List date range covered. Flag any record missing a field that Task 1 marked as required.

2d. Tradition reflection (Go Deeper) inventory

This section has additional analysis requirements. For every tradition reflection across all journeys and days, report completeness across all field types — not just text fields.

Apply the universal field-type completeness rules from the Reporting Depth section above to every tradition reflection field. Flag any record where one or more fields of any type are missing — including no era selected, no theme tagged, no authorType chosen, even if the text content (quote and summary) is present. A partially complete record is a data quality issue regardless of which fields are missing. Tradition reflections with no journey or day association are orphans — flag them in a separate list.

Then produce two additional summary views:

  • Thinker usage map: Group all TRs by author name. Flag any author appearing in more than one journey. Flag any author appearing more than twice across the whole content set.
  • Tradition type breakdown: Count TRs by authorType. Show count and percentage for each type. Flag any type with zero entries.

2e. TTS coverage audit — published records only

TTS audio is billed by character count. Before any decision is made about generating or commissioning TTS, we need to know exactly how many characters of text content exist without audio coverage, and where it lives.

Query published records only — exclude any document whose _id starts with drafts.

Do not use judgment to determine which fields are TTS candidates. Use the schema. A TTS-candidate field is defined as: a text field that has a corresponding audio upload field adjacent to it in the same schema object, excluding any field that is part of the auditio object (which holds curated music, not TTS). Use the field inventory from Task 1 to identify all such text+audio pairs across every schema. This grounds the audit in the architecture — if a text field was designed with a paired audio field, TTS was intended. If the audio field is empty on a published record, that is the gap.

For each TTS-candidate text field on each published document, report:

  • Whether an audioFile is already attached to that record (yes/no)
  • Character count of the text field content
  • Word count of the text field content

Then produce two aggregations:

  • By document type: Total characters and words missing TTS coverage, grouped by journeyDay / dailyPrompt / traditionReflection / contentItem
  • Grand total: All characters and words across all published documents with no audio. This is the number Sheri uses to estimate cost.

Include a cost reference table using current published pricing for the three most relevant TTS providers. Do not calculate a specific dollar figure — output the character total and let Sheri apply the rate. The table should show rate per 1,000 characters for: Google Cloud TTS Neural2, Amazon Polly Neural, and ElevenLabs (lowest available tier). Verify current pricing via web search before including figures — do not use potentially outdated training data for pricing.

2f. Missing content flags — consolidated red list with recommendations

After completing 2a-2e, produce a single consolidated list of every missing or partial field across all document types. Sort by severity: required fields missing first, optional fields missing second, partial fields last. TTS gaps from 2e are a separate category at the bottom.

For each item in the red list, include a brief recommendation: is this a Sanity entry task (Sheri enters missing content), a schema task (field needs to be added, removed, or restructured), or a data migration task (content exists but is in the wrong field and needs to be moved)? Distinguishing these three types means Sheri immediately knows whether to open Sanity Studio, schedule a CC session, or flag it for the schema design session.

Task 3 Content Dashboard — Live Next.js Page Priority 2

Build the content dashboard as a protected page in the existing Next.js app at /dashboard. The page queries Sanity server-side at load time so it is always current without any manual step, script, or webhook. Sheri bookmarks it and opens it before any content session.

Why not a script or webhook
A script requires a manual trigger and produces stale data between runs. A webhook requires a live endpoint, separate hosting, and handling for missed events. A Next.js server component uses infrastructure already in place — the app, the existing Sanity client, and Vercel — and is always current on load with no additional infrastructure.

Route and implementation

app/dashboard/page.tsx — server component. Styled with Tailwind using the existing KALLOS espresso/parchment/gold design system. Add an environment variable gate (DASHBOARD_ENABLED=true) so the route returns 404 if the variable is absent — this protects against accidental public exposure at launch. Set revalidate: 60 so repeated page loads within a session serve a cache rather than hammering Sanity, while staying no more than 60 seconds stale.

Data fetching

One GROQ query per section, all run server-side. Add named query functions to lib/sanity.ts (e.g., getJourneyCompletionStatus, getTraditionReflectionInventory, getAudioStatus). Use the existing Sanity client. No new dependencies required.
