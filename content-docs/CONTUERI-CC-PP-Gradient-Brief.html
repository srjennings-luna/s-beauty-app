Stack

Next.js 14 (App Router), TypeScript, Tailwind CSS, Sanity CMS. Deployed on Vercel. Currently web-only (mobile-first web app). Native iOS and Android are planned for a future phase — this matters for the architectural decisions below.

Design system context

The app has two visual modes:

  • Espresso mode (#16110d) — immersive screens: P&P, Journey steps, Visio Divina, Breathe
  • Parchment mode (#fdf6e8) — browse/reading screens: Today landing, Explore, Journeys list, Library

No rounded corners anywhere. Full design system in CLAUDE_REFERENCE.md.

What prompted this work

The Pause & Ponder (P&P) daily screen features a full-bleed classical artwork as hero image. The existing design is clean but undifferentiated — all P&P screens look identical regardless of content type. During a design exploration session (May 2026), we developed a system using classical fresco pigment hues as ambient gradient backgrounds, one per content type. The goal: each P&P type has a distinct atmosphere the user feels even if they can't articulate why — similar to how Spotify extracts album art color into the player UI.

BG-01 closed: A previous UX backlog item (BG-01) tracked a potential parchment-mode gradient to reduce the visual seam between the espresso splash and flat parchment screens. That item is now closed — won't do. The P&P gradient system resolves the seam question differently: espresso P&P screens become distinctly rich, while parchment screens staying flat is now intentional (it signals "browsing mode" vs. "contemplative mode"). No gradient should be added to Explore, Today, Library, or Journeys screens.

2. What to Build

The feature in one sentence

Each P&P screen gets a three-layer ambient gradient fixed to the viewport (so content scrolls over it), with gradient color and accent color both driven by contentItem.contentType.

Visual behavior

  • Three radial-gradient ellipse layers, absolutely positioned inside a position: fixed container
  • Upper-left sweep (large, 22s drift cycle), diagonal band (rotated ellipse, 18s), lower-right pool (28s)
  • Each layer animates with transform: translate() + opacity — GPU-accelerated, no jumping. Never animate the background property directly.
  • Travel distance: 70–80px for sweep and pool, 60px for band. Opacity range: 1.0 → 0.55 at peak drift.
  • Independent cycle durations (22s / 18s / 28s) so layers never synchronize — creates organic, non-repeating movement.
  • The gradient sits behind all content; nav and scrollable content layer above it.

Accent color cascade

The same hex color used for the gradient is used as the accent throughout the P&P screen: section labels (READING, ACTION), READ MORE / LISTEN buttons, scripture attribution lines, checkbox borders, active tab indicator. This makes every interactive and label element feel like it belongs to that content type's atmosphere.

4. Architectural Decisions

Claude Code was consulted on the implementation approach before writing this brief, specifically regarding the future iOS/Android migration. These decisions reflect that consult. Do not deviate without discussion.

4a. TypeScript as color token source of truth

Create lib/ppGradients.ts as a pure TypeScript file with no web-specific imports (no CSS, no document). This file will be the only place colors are defined — it survives the eventual React Native migration unchanged.

// lib/ppGradients.ts
export const PP_GRADIENTS = {
  'sacred-art':    { hex: '#7ba2b8', rgb: '123,162,184' },
  'music':         { hex: '#9a8a9e', rgb: '154,138,158' },
  'literature':    { hex: '#c9a07c', rgb: '201,160,124' },
  'landscape':     { hex: '#a8ae9a', rgb: '168,174,154' },
  'food-wine':     { hex: '#c68a77', rgb: '198,138,119' },
  'math-science':  { hex: '#84a9a2', rgb: '132,169,162' },
  'thinker':       { hex: '#E4C371', rgb: '228,195,113' },
  'watch-listen':  { hex: '#b8869a', rgb: '184,134,154' },
  'photography':   { hex: '#a49898', rgb: '164,152,152' },
} as const satisfies Record<ContentType, { hex: string; rgb: string }>;

export type PPGradientType = keyof typeof PP_GRADIENTS;

export function getPPGradient(contentType: ContentType) {
  return PP_GRADIENTS[contentType as PPGradientType] ?? PP_GRADIENTS['sacred-art'];
}

4b. One wrapper component with CSS custom property cascade

Create components/PPGradientBackground/index.tsx. This component accepts contentType and children. It writes --pp-accent and --pp-accent-rgb to the wrapper element as CSS custom properties so every child element in the P&P screen can reference them without receiving color as a prop.

// components/PPGradientBackground/index.tsx
import { getPPGradient } from '@/lib/ppGradients';
import type { ContentType } from '@/lib/types';
import styles from './gradient.module.css';

interface Props {
  contentType: ContentType;
  children: React.ReactNode;
}

export default function PPGradientBackground({ contentType, children }: Props) {
  const token = getPPGradient(contentType);
  return (
    <div
      className={styles.phone}
      style={{ '--pp-accent': token.hex, '--pp-accent-rgb': token.rgb } as React.CSSProperties}
    >
      <div className={styles.gradientBg} aria-hidden="true">
        <div className={styles.gSweep} />
        <div className={styles.gBand} />
        <div className={styles.gPool} />
      </div>
      {children}
    </div>
  );
}

Child elements reference the accent color via color: var(--pp-accent) in CSS. No hex strings should appear outside lib/ppGradients.ts.

4c. CSS module for gradient styles

Put all gradient CSS (keyframes, layer sizing, animation) in components/PPGradientBackground/gradient.module.css. Use var(--pp-accent-rgb) inside rgba() for gradient colors so the single custom property drives all three layers automatically.

/* gradient.module.css — key patterns */

.gradientBg {
  position: fixed;
  top: 0; left: 50%; transform: translateX(-50%);
  width: 345px; height: 100dvh;   /* 100dvh not 100vh — see mobile web note */
  background: #16110d;
  overflow: hidden;
  pointer-events: none;
  z-index: 0;
}

.gSweep {
  position: absolute;
  width: 380px; height: 460px;
  top: -140px; left: -110px;
  border-radius: 50%;
  will-change: transform, opacity;   /* GPU layer promotion */
  background: radial-gradient(ellipse at center,
    rgba(var(--pp-accent-rgb), 0.32) 0%,
    rgba(var(--pp-accent-rgb), 0.10) 40%,
    transparent 68%);
  animation: sweepDrift 22s ease-in-out infinite;
}

/* similar for .gBand (rotated ellipse, 18s) and .gPool (lower-right, 28s) */

@keyframes sweepDrift {
  0%   { transform: translate(0px, 0px);   opacity: 1; }
  40%  { transform: translate(70px, 80px); opacity: 0.55; }
  75%  { transform: translate(30px, 40px); opacity: 0.80; }
  100% { transform: translate(0px, 0px);   opacity: 1; }
}

@media (prefers-reduced-motion: reduce) {
  .gSweep, .gBand, .gPool { animation: none; }
}

4d. No cross-platform abstraction layer — plan for a clean rewrite later

Do not add Expo, Tamagui, NativeWind, or any React Native abstraction layer to the current Next.js codebase. The web and native gradient implementations will necessarily differ (CSS radial gradients don't exist in React Native; the RN approach will use @shopify/react-native-skia or expo-linear-gradient + react-native-reanimated). The correct approach is to keep the component interface clean now so the internals can be swapped without touching the consuming code. The migration surface area when going native is exactly components/PPGradientBackground/ — nothing else.

4b. Related Journey CTA Color

The P&P screen can surface a "Related Journey" CTA linking users to a companion journey. This CTA should keep the app's standard sacred gold (#C19B5F) — it does not adopt the type accent color.

Rationale: the Related Journey CTA is the only outbound action on the screen. Sacred gold already carries "primary action" meaning throughout the app. Keeping it gold signals "this takes you somewhere else" — it acts as a deliberate contrast point within the monochromatic type system. If every interactive element adopts the type color, the CTA loses hierarchy. Sacred gold as the exit color is semantically correct.

This will be revisited after the gradient system is implemented and reviewed in production. No change to the CTA styling is needed in this build.

5. Mobile Web Requirements

Three specific issues to address — all confirmed by CC architectural consult:

5a. Use 100dvh not 100vh

On iOS Safari, when the virtual keyboard appears, position: fixed elements can jump or repaint if sized with 100vh. Use 100dvh (dynamic viewport height, supported Safari 15.4+) on the gradient container. This handles the keyboard appearance correctly.

5b. Add will-change: transform, opacity to animated layers

Three simultaneously-animated layers can cause jank on budget Android devices (2GB RAM). Adding will-change: transform, opacity promotes each layer to its own GPU compositing layer upfront rather than on first animation frame. Apply to .gSweep, .gBand, and .gPool.

5c. Respect reduced motion

Wrap all animations in a @media (prefers-reduced-motion: reduce) block that disables them. The gradient should still display (static is fine) — just no movement. This is both an accessibility requirement and a battery consideration.

Desktop / iPad fallback needed. The width: 345px fixed container assumes a narrow phone viewport. On an iPad or desktop browser, the gradient will render as a narrow strip against the page background. Either cap the component to phone-width screens only (media query) or implement a responsive fallback (e.g. center the gradient in a max-width: 480px container). Confirm approach with Sheri before implementing if unclear.

6. Where to Plug This In

Data path

The P&P page (app/prompt/PromptClient.tsx) already fetches the dailyPrompt document from Sanity, which includes a reference to contentItem. The contentType field lives on contentItem.contentType. The GROQ query already resolves this — confirm by checking lib/sanity.ts for the getDailyPrompt / getPromptByDate query. If contentItem.contentType is not currently projected, add it.

Wrapping PromptClient

Wrap the existing PromptClient content with PPGradientBackground:

// app/prompt/PromptClient.tsx (simplified)
import PPGradientBackground from '@/components/PPGradientBackground';

export default function PromptClient({ prompt }) {
  return (
    <PPGradientBackground contentType={prompt.contentItem?.contentType ?? 'sacred-art'}>
      {/* existing prompt content */}
    </PPGradientBackground>
  );
}

Updating existing accent color references

The P&P screen currently uses hardcoded accent colors (sacred gold #C19B5F and blue mist #8aacb8 in various places). These should be migrated to reference var(--pp-accent) so the type-specific color cascades correctly. Key elements to update:

  • Section labels (.section-label, READING, ACTION etc.)
  • READ MORE / LISTEN button text
  • Scripture attribution line
  • Checkbox border (actio-row checkbox)
  • Active tab indicator in the P&P tab bar

7. What NOT to Touch

  • Explore page — light parchment background. No gradient. The 8 fresco colors are for espresso mode only. Theme cards use their own theme.color field from Sanity — a completely separate system.
  • Today landing, Library, Journeys list — flat parchment. BG-01 (parchment gradient) is closed. See Section 1.
  • Journey step gradient panels — Journey steps have their own plum gradient panel treatment on the Context block. This is a separate, unrelated system. Do not merge or override.
  • ArtworkViewer — neutral overlay. No type-based color treatment here.
  • Splash screen — has its own Whisper gradient (mauve-tinted espresso). Leave it.
