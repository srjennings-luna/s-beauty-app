Decisions Made — Do Not Relitigate

R5 — journeyDay promotion: Promote journeyDay from inline object to standalone document schema. Migrate now while there are only 25 days across 5 journeys. Rationale: removes an architectural ceiling at low cost. If cross-journey day reuse is never needed, nothing is lost. If it is needed, it is already supported.
R1 — Three fields, three jobs (Option C): Rename contentItem.curatorNote to contentItem.artworkHook. Each field has one job and one context:

contentItem.artworkHook — surprising fact about this specific artwork or piece. Safe to display anywhere the content item appears (Journey, Explore, Library, P&P). Never journey-specific.
dailyPrompt.curatorNote — the hook for this specific P&P day. Shown only on P&P. Can differ entirely from the artwork hook.
journeyDay.encounterNote — the journey-specific interpretive layer. Shown only in Journey context. Already well-named, no change needed.

Task 1 Promote journeyDay to standalone document schema R5

Currently journeyDay is an inline object inside journey.days[]. This task promotes it to a first-class document with its own _id, makes it independently queryable, and migrates the 25 existing inline day objects to standalone documents.

1a. Create the new journeyDay schema

In sanity/schemaTypes/journeyDay.ts (new file): define journeyDay as a document type with all existing inline fields preserved exactly — no field renames here, just the type promotion. Add one new field: journey (reference to journey document, required). Register in sanity/schemaTypes/index.ts.

1b. Update journey schema

In sanity/schemaTypes/journey.ts: change days from array of journeyDay objects to array of references to journeyDay documents. Keep the field name days.

1c. Write and run the data migration script

Write a migration script at scripts/migrate-journeydays-to-documents.ts. For each journey, for each inline day object in journey.days[]:

  1. Create a new journeyDay document with all field values copied from the inline object
  2. Set the new document's journey reference to the parent journey's _id
  3. Replace the inline object in journey.days[] with a reference to the new document's _id

Run against the production dataset. Verify: after migration, *[_type == "journeyDay"] should return 25 documents. Each parent journey's days array should contain 25 total references across all journeys.

1d. Update GROQ queries

In lib/sanity.ts: update every query that accesses journey days. The traversal pattern changes from inline projection (days[]{ ... }) to reference resolution (days[]->{ ... }). Update all 5 named query functions that touch journey day fields. Test each query returns the same shape as before.

1e. Update TypeScript types

In lib/types.ts: JourneyDay gains _id: string and journey: { _ref: string }. No other type changes needed for this task.

1f. Update seed scripts

Check all files in scripts/ that create journey or journey day records. Update to use the new document creation pattern rather than inline object insertion.

Migration risk

Task 2 Rename curatorNote → artworkHook + field audit R1

This task has two parts: the schema/code rename, and an interactive audit of existing content to determine what to do with the 47 records that have content in the old field.

2a. Schema rename

In sanity/schemaTypes/contentItem.ts:

  • Rename field curatorNote → artworkHook
  • Rename field curatorNoteAudio → artworkHookAudio
  • Update the field title and description in the Studio so it reads: "Artwork Hook — The surprising fact about this specific piece. Safe to show anywhere this piece appears. Must be about the artwork itself, not a journey or day context."

In lib/sanity.ts: update all GROQ projections that reference curatorNote or curatorNoteAudio on contentItem. In lib/types.ts: rename the field on the ContentItem type. In all components: update any reference to content.curatorNote.

2b. Interactive content audit — pause here and work with Sheri

Query all 47 content items that have a non-empty curatorNote (now renamed artworkHook in schema, but still old data). For each, display:

  • Content item title and type
  • First 150 characters of the curator note content
  • Which journey(s) it appears in

Present this list to Sheri 5 items at a time. For each item, ask: "Is this about the artwork itself (keep as artworkHook), or was it written for a specific journey context (flag for rewrite)?"

Tag each record as one of:

  • KEEP — artwork-specific, safe as artworkHook
  • REVIEW — journey-specific or unclear, flag for Sheri to rewrite later

Do not delete or overwrite any content. REVIEW-tagged items keep their current content but get a Sanity Studio note added. Produce a summary list of all KEEP / REVIEW decisions at the end.

Output

Schema updated. GROQ, types, and components updated. Content audit complete with KEEP/REVIEW list saved to content-docs/KALLOS-artworkHook-Audit.html. No content deleted.

Task 3 Add new fields — auditio genre + split audio source

Additive schema changes only. No data migration required — existing records simply have these fields empty until Sheri fills them in Sanity.

3a. Auditio genre field

Add a genre field to the auditio object in both dailyPrompt.ts and the new journeyDay.ts schema. Type: string with a predefined list of options (single-select). Options:

  • gregorian-chant
  • sacred-polyphony
  • classical-orchestral
  • contemporary-sacred
  • ambient-meditation
  • folk
  • world-music
  • jazz
  • secular

Field title in Studio: "Genre" with description "Select the closest genre. Used to track variety across journeys and prevent over-reliance on any single register."

3b. Split audio source fields

The current contentItem.audioSource pattern combines composer/artist and work title in a single string. Add two new optional fields to the auditio object on both dailyPrompt and journeyDay schemas:

  • composerArtist (string) — the composer or performing artist name only
  • workTitle (string) — the specific piece, track, or work title only

Keep the existing source/attribution string field — do not remove it. These two new fields are additive. Existing records keep their single string; new entries use the split fields. The dashboard and future content intelligence queries will prefer the split fields when present.

3c. Update TypeScript types and GROQ

Add genre?: string, composerArtist?: string, workTitle?: string to the relevant TypeScript types. Add to the auditio projection in GROQ queries. No component changes required — these fields are dashboard/audit use only for now.

Task 5 Update dashboard for new journeyDay structure

Post-R5, journeyDays are standalone documents. The dashboard Section 1 (Journey Completion Status) was built to traverse from journey to inline days. Update it to query journeyDays directly.

5a. Update getJourneyCompletionStatus query

Replace the inline traversal with a direct journeyDay query: *[_type == "journeyDay"] grouped by journey._ref. This is faster and cleaner than the original. Update the named function in lib/sanity.ts and any types it returns.

5b. Add genre distribution to dashboard Section 4

Now that the genre field exists on auditio (Task 3), add a genre distribution view to dashboard Section 4 (Audio Status). Show: count of records per genre across all journey days and P&P prompts. Flag any genre that represents more than 40% of all auditio selections.

5c. Add artworkHook audit flag to dashboard

Add a small indicator in the Content Item section of the dashboard: how many content items have been tagged REVIEW in the artworkHook audit (from Task 2b). This gives Sheri a reminder that there are items needing rewrite without requiring her to remember.
