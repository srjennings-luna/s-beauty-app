What this is. A new read-only content review tool for Sheri, added as a subroute of the existing /dashboard. It lets her see KALLOS content fields in a configurable grid: all rows, all columns, all togglable. The interaction model is a configurable data grid with column show/hide, row filter and pin, cell expansion, and named preset views. This pattern comes from professional seed-stage analytics dashboards where non-technical users need to scan large content sets across many dimensions simultaneously and configure their view without developer help. Phase 1 is read-only. No editing, no writes, no annotations.

Key patterns to replicate from these screenshots:

  • Column toggle panel: a dropdown flyout with checkboxes, a search field at the top for filtering the column list, and a Reset button at the bottom. Selected columns highlighted in the accent color.
  • Quick filter chips (All / Mine / Pinned) as a horizontal chip row above the table. For KALLOS: All / journeyDay / dailyPrompt / Pinned / Voice flags.
  • Color-coded status badges (On-Time in green, Ahead in blue, Behind in red). KALLOS equivalent: draft/published indicator on each row.
  • Sortable column headers with directional arrows on active sort column.
  • Inline cell expansion: row grows in place, full text visible, surrounding rows undisturbed. Click to expand, click again or press Escape to revert.

GROQ function spec

Add to lib/sanity.ts under a new // Review Grid section comment. Returns a typed union of journeyDay and dailyPrompt rows with all text fields and pre-joined ref fields. One round-trip.

export async function getReviewGridRows() {
  return sanityClient.fetch(`{
    "journeyDays": *[_type == "journeyDay"] | order(journey->title asc, dayNumber asc) {
      _id,
      _type,
      dayNumber,
      dayTitle,
      "journeyTitle": journey->title,
      "journeySlug": journey->slug.current,
      openText,
      "openTextAudioUrl": openTextAudio.asset->url,
      encounterGuidance,
      encounterNote,
      "encounterNoteAudioUrl": encounterNoteAudio.asset->url,
      "artworkTitle": encounterContent->title,
      "artworkHook": encounterContent->artworkHook,
      "artworkHookAudioUrl": encounterContent->artworkHookAudio.asset->url,
      "context": encounterContent->context,
      "contextAudioUrl": encounterContent->contextAudio.asset->url,
      "imageUrl": encounterContent->image.asset->url,
      lectio,
      auditio,
      reflectionQuestions,
      "reflectionQuestionsAudioUrl": reflectionQuestionsAudio.asset->url,
      connectThread,
      "goDeeperTitles": goDeeper[]->{_id, title}
    },
    "dailyPrompts": *[_type == "dailyPrompt"] | order(date desc) {
      _id,
      _type,
      date,
      promptQuestion,
      curatorNote,
      "artworkTitle": contentItem->title,
      "imageUrl": contentItem->image.asset->url,
      context,
      lectio,
      auditio,
      actio,
      verbaOriginal
    }
  }`)
}

Return shape is { journeyDays: JourneyDayRow[], dailyPrompts: DailyPromptRow[] }. The client merges these into a single display array keyed on _id and branching on _type. TypeScript types: discriminated union GridRow = JourneyDayRow | DailyPromptRow with _type as the discriminant. Do not use a merged type with all fields optional.

Preset registry spec

File: app/dashboard/review/presets.ts. Each preset defines its default column set and the one to three parameters the user picks. Preset selection sets URL param preset=id. The user can override the column set by toggling columns after selecting a preset; overrides add to URL as cols=.

idlabeldefaultColumnsparameter
recordRecord viewAll fields, sequential orderid (which record, populated by clicking identifier link)
fieldField comparisonidentifiers + 1 focus fieldfocusField (dropdown of all column keys), docType (journeyDay / dailyPrompt / both)
arcJourney arcidentifiers + openText + connectThread + auditio titlejourney (dropdown of all journey slugs)
parallelParallel entry pointidentifiers + openText + encounterNote + lectio philosophydayNumber (1-8, shows that day across all journeys)
pairingContent pairingidentifiers + two chosen fieldsfieldA and fieldB (two dropdowns from column key list)
customCustomIdentifiers only, all other columns offNone. User builds from blank.

Identifier columns are always present and always pinned left. They are never togglable off. For journeyDay: Journey Title + Day label (formatted "Day N: {dayTitle}"). For dailyPrompt: Date + Artwork Title. The identifier columns are visually distinct (slightly darker background). The first identifier column is a link to /dashboard/review/record/[id].

Default sort. journeyDay rows: Journey Title ascending, then Day Number ascending. dailyPrompt rows: Date descending. All journeyDay rows appear before all dailyPrompt rows in mixed views. User-selected column sort overrides this default; a reset button restores it.

Voice rules config spec

File: app/dashboard/review/voiceRules.ts. Each rule is checked against every text cell on render. Matching cells get a small warning badge (amber dot, count of matches on hover). A "Voice flags" quick-filter chip at the top of the grid narrows rows to those with at least one match.

  id: string
  pattern: RegExp
  label: string
  severity: 'error' | 'warn'
  suggestion: string
}

Task 5 CSV export and voice rule scanner

CSV export

Create app/dashboard/review/export/route.ts. The route handler accepts the same URL params as the grid (preset, docType, journey, cols, etc.), re-runs getReviewGridRows(), applies the same filter logic, and returns a UTF-8 CSV file download. Column headers match the column labels in the grid. Long text fields are quoted and newlines within cells are escaped. Add a "Download CSV" button to the grid toolbar that links to this route with current URL params appended.

Voice rule scanner

Create app/dashboard/review/voiceRules.ts with the config from the spec above. In ReviewGridClient.tsx, on render of each text cell, run all patterns against the cell content. If any match: show a small amber indicator badge on the cell (e.g. a dot with the match count). Hovering the badge shows a tooltip listing which rules matched and the suggestion for each.

Add a "Voice flags" quick-filter chip to the filter bar. When active (voiceFlags=1 in URL), the grid shows only rows that have at least one voice rule match in any visible column. This lets Sheri run a full content audit for banned words and em dashes in one view.

Output
"Download CSV" button downloads a well-formed CSV of the current grid view. Voice rule badges appear on matching cells. "Voice flags" filter narrows grid to flagged rows only. Em dash rule catches all Unicode em dashes (U+2014).

Constraints

  • Phase 1 is read-only. No editing, no saving, no Sanity write operations of any kind.
  • All interactive state lives in URL params. No useState for anything that should survive a page reload or be shareable.
  • No em dashes in code you write: not in code comments, not in string literals, not in JSX text. This rule applies to the code Claude produces, not to Sanity content data returned by GROQ queries. Do not modify, strip, or correct em dashes in content data. The voice scanner surfaces them for Sheri to fix in Sanity Studio; that is its entire job.
