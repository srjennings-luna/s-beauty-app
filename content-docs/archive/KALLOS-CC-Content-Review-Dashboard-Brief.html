Background — What Exists Today

KALLOS has a live operational dashboard at /dashboard in the Next.js app (deployed to Vercel, gated by DASHBOARD_ENABLED=true env var). It is a server-rendered page with seven sections covering journey completion status, content library, tradition reflections, audio gaps, schema health, TTS coverage, and companion journey placeholders. It is read-only, uses named GROQ query functions in lib/sanity.ts, and revalidates every 60 seconds. See app/dashboard/page.tsx for full implementation.

The current dashboard is an operational tool: it surfaces what is missing, what is duplicated, what has audio gaps, what needs attention. It does not evaluate content quality or allow editorial comparison across records.

The content types relevant to this brief are dailyPrompt (Pause and Ponder daily prompts, currently 27 records) and journeyDay (standalone documents, 25 records across 5 journeys, promoted from inline objects in April 2026). Both schemas share the Auditio object structure. See CLAUDE.md Schema Status section for full field inventory.

The Editorial Problem

KALLOS has a set of editorial principles defined in CLAUDE.md and two content guides (P&P and Journey). These exist as prose rules. There is currently no way to look at actual content across records and say "this example follows principle 3 and 6, this one doesn't, here is why." Standards are defined but not demonstrated. Quality is evaluated one record at a time inside Sanity Studio, not comparatively.

As content scales toward hundreds of records, Sheri needs a tool that lets her:

  • See content fields in the sequence the user experiences them (not in the Sanity document form order)
  • Compare the same field across many records at once to identify what good looks like
  • Look at a full journey arc to evaluate whether content develops across 7 days or repeats itself
  • Control the view precisely: show only the fields relevant to the current audit, hide the rest

This is distinct from the operational dashboard. That tool asks "what is missing?" This tool asks "is what is there good?"

What We Researched

Before designing a custom build, Sanity's native ecosystem was surveyed. Summary of findings:

OptionWhat it doesGap
Sanity document comparison tool (native, Spring 2025) Side-by-side view of two versions of the same document Version history only, not cross-record comparison
Bulk Actions Table plugin (contentwrap, actively maintained) Table view of one document type with column show/hide, sort, search, bulk publish/unpublish One document type at a time; no cross-type rows; no sequential content flow; no export
CSV export (GUI) Does not exist Still an open feature request (GitHub issue 4660, no milestone). CLI + jq workaround is developer-only.

Sanity App SDK (Spring 2025) React hooks toolkit for building custom apps inside Studio, with live data, built-in auth, caching Not a ready-made tool; a build platform. Promising for a Studio-embedded version of this feature.
Agent Actions / Content Agent (Spring 2025) AI-driven content audit, field-level updates, automated gap detection Automation tool, not a human review interface

Conclusion: no off-the-shelf solution covers this use case. Custom build is required.

Interaction Model — What Sheri Decided

The following interaction model was established in the design discussion. This is the spec for what needs to be built. Architecture decisions are open (see next section).

Core model: configurable grid

Rows are records (journeyDays, dailyPrompts, or both). Columns are fields. All fields are accessible; none are eliminated by design. Sheri controls what is visible through column and row toggles. This means the named "views" below are not separate screens; they are saved column/row configurations on a single grid.

Row controls

  • Filter by document type (journeyDay / dailyPrompt / both)
  • Filter by journey (BTG, Myth, Light, Bosch, etc.)
  • Filter by date range (for dailyPrompts)
  • Filter by completeness (days with missing fields)
  • Select specific rows to pin for focused comparison; hide unselected rows
  • Sort by any column (ascending / descending)

Column controls

Every text-content field should be available as a column. All toggled on by default; any can be turned off. Fields include (but are not limited to):

openText artworkHook / curatorNote context promptQuestion encounterNote lectio (philosophy) lectio (scripture) auditio title + artist reflectionQuestions connectThread goDeeper (titles) actio image thumbnail dayNumber date

Long text handling

Most content fields are long text that will not fit in a grid cell. Two modes are needed: a truncated cell view (2-3 lines, expand toggle per cell) for scanning, and a detail panel for reading. The detail panel opens when a row is selected and shows all fields for that record in the sequence the user experiences them. This is the "record view" and "full read-through" in one place.

Named preset views (column/row configurations, not separate screens)

Record view
One row selected, all columns visible, detail panel open. Shows the complete sequential experience for a single P&P or journey day. Primary use: approve content before publishing, or audit an existing record as a unit.
Field comparison view
All rows, one or two columns visible. For example: all curator notes across all P&Ps, or all opening texts across all journey days. Primary use: standard-setting, identifying what good looks like for a specific field across many records.
Journey arc view
One journey filtered (all 7 days), structural columns visible (openText, connectThread, auditio title). Primary use: evaluate whether the emotional arc develops across the week, whether music changes register, whether connect threads actually build or just repeat.
Parallel entry point view
Day 1 rows filtered across multiple journeys, key columns visible. Primary use: verify that all journey entry points are distinct in tone and approach.
Content pairing view
Custom column selection: for example, lectio (philosophy) + lectio (scripture) + auditio title together, or reflectionQuestions + connectThread + image. Primary use: evaluate how specific fields work as pairs or groups, not in isolation.

TTS view (future, not Phase 1)

Sheri named a fifth future view she called "really listen": playing the TTS audio for each field in sequence, experiencing the content as audio rather than text. This is time-based and fundamentally different from the grid. Defer entirely. No architecture decisions needed now.

Comments / annotations (future, not Phase 1)

Annotation was discussed and deferred deliberately. The core problem: a comment written while comparing 20 curator notes means something different than the same comment written on a record in isolation. Capturing that context is non-trivial. Phase 1 is read-only. The architecture should not preclude comments being added later, but no design or data model decisions are needed now.

Editorial tagging (future, not Phase 1)

Sheri's longer-term goal is to tag records with which editorial principles they exemplify ("this opening text demonstrates principles 1, 4, and 6"), and eventually to have automated similarity detection surface candidates for her to accept, reject, or relabel. Phase 1 does not include any tagging or write capability. The architecture should not preclude it. See "Future features to plan around" section below.

Architecture Questions for Opus

The following questions are open. Sheri and Cowork have not resolved them. Opus is being asked to recommend an approach for each, with reasoning.

Q1 Where should this live: extend the existing /dashboard, or build as a Sanity App SDK tool?

The existing /dashboard in the Next.js app is half-built: it has GROQ queries, Sanity client access, and is already deployed. Extending it is lower effort. But it lives outside Studio, which means a context switch for Sheri (editing in Studio, reviewing in a separate Vercel URL).

Sanity's App SDK (Spring 2025) allows building custom React apps that run inside Studio as embedded tools. They have built-in authentication, live data with caching, and React hooks for querying content. Building there would mean the comparison view lives alongside the editing experience, in the same window.

Considerations Opus should weigh:

  • The App SDK is relatively new (Spring 2025). How mature is it? Is it stable enough for a production tool, or is it still moving fast?
  • How much harder is the App SDK build vs. extending the Next.js dashboard? The existing dashboard is a known quantity.
  • Is a hybrid sensible: build in Next.js first, then migrate the comparison view into the App SDK once the design is stable?
  • The Presentation plugin is already installed and working. Does the App SDK interact with or conflict with the Presentation setup?
Q2 How do you handle cross-document-type rows in a single grid?

journeyDay and dailyPrompt have overlapping but not identical schemas. A journeyDay has openText, encounterNote, connectThread; a dailyPrompt has promptQuestion, curatorNote, actio. They share lectio, auditio, and image structure.

When both types appear as rows in the same grid, some columns will be empty for some rows. That is acceptable and by design (Sheri wants all fields accessible, empty is fine). But the GROQ query strategy and the TypeScript type for a "row" need to handle this gracefully.

Questions for Opus:

  • What is the cleanest GROQ strategy for fetching a unified row set across two document types with partial schema overlap?
  • Should the row type be a union type, a merged type with all fields optional, or something else?
  • Are there field name collisions to handle? (curatorNote on dailyPrompt vs. artworkHook on contentItem are different things with similar names. Mapping needs to be explicit.)
Q3 Where does the "record view" detail panel live, and how is it triggered?

The detail panel shows all fields for one record in sequential order (the order the user experiences them in the app, not the Sanity form order). It needs to open from the grid without navigating away, and it needs to display fields in a specific sequence that is different from the schema order.

Questions for Opus:

  • Slide-over panel, modal, or expandable row? What works best for long-form content review?
  • The sequential order for journeyDay is: image, openText, artworkHook (from linked contentItem), context (from linked contentItem), lectio, auditio, reflectionQuestions, connectThread, goDeeper titles. Several of these require traversing a reference (the linked contentItem). Does this create a second GROQ fetch, or should the main grid query pre-join these via projection?
  • What is the right GROQ projection to pre-fetch contentItem fields for journeyDay rows without over-fetching?
Q4 How should preset views be saved and restored?

The named views (record view, field comparison view, arc view, etc.) are column/row configurations that should be quickly accessible. Sheri should be able to switch between them without re-configuring each time.

Questions for Opus:

  • URL params, localStorage, or hardcoded named presets as buttons? Each has tradeoffs. URL params are shareable and bookmarkable. localStorage persists without a server. Hardcoded presets are zero-config but inflexible.
  • Should the user be able to save custom presets, or are the named presets above sufficient for Phase 1?
  • How does this interact with the filter state (which journey is selected, which rows are pinned)?
Q5 What is the right data fetching strategy for a configurable grid at this content volume?

Current content volume: 25 journeyDay documents, 27 dailyPrompt documents. Growing toward hundreds of records over the next 12 months. All fields per record. Some fields require ref traversal (journeyDay references contentItem for artworkHook, context, image).

Questions for Opus:

  • Is it practical to fetch all records with all fields in a single GROQ query with projections, or is the payload too large?
  • Should the grid load all rows eagerly, or paginate / virtualize?
  • Does the existing revalidate: 60 strategy work for this view, or does the comparison use case need live updates?
  • The existing dashboard uses server components with named GROQ functions in lib/sanity.ts. Should the comparison view follow the same pattern, or does the interactive filtering require client-side data with a different fetch approach?

Future Features to Plan Around (Not Phase 1)

These are not being built now, but the Phase 1 architecture should not accidentally prevent them. Opus should flag any Phase 1 decision that would create technical debt against these future features.

Editorial tagging

Sheri's intention: tag records in the grid with which editorial principles they exemplify. First pass is manual (she applies tags). Second pass introduces automated detection with an accept/reject/modify workflow. The tags need to persist somewhere. Options discussed: back into Sanity as fields on records, a separate annotation store (JSON in the repo), or dashboard-only state. No decision made yet. The Phase 1 architecture should not assume tags will not exist.

Accept/reject/modify for automated similarity detection

The current operational dashboard already does work-level repeat detection (same composerArtist + workTitle used twice). The longer-term vision is semantic similarity detection across content fields: two pieces exploring the same theological territory, two opening texts using the same structural move. The human review workflow is: system surfaces a candidate, Sheri accepts the tag, rejects it, or renames it. Where do those decisions persist? The Phase 1 architecture should not foreclose a lightweight annotation layer.

CSV / spreadsheet export

Sanity does not have a native GUI export. A "Download this view as CSV" button on the comparison grid would let Sheri work with content in Numbers or Excel for ad hoc analysis. This is a natural extension of the grid once the data model is established. Plan for it; do not build it yet.

TTS "really listen" view

A separate time-based view that plays TTS audio for all fields in sequence, simulating the in-app experience as audio. Entirely separate from the grid. Defer; no architecture decisions needed now.

Constraints

  • Sheri does not write code. The tool must work in a browser, with no terminal interaction required.
  • Phase 1 is read-only. No editing, no saving, no write operations of any kind.
  • The existing operational dashboard at /dashboard must continue working. Any new build is additive, not a replacement.
  • No em dashes in any content or code comments. This rule applies everywhere in KALLOS.
  • Build verification: npx tsc --noEmit from repo root (SWC binary not available on Cowork VM). Commit after every significant step.
  • The comparison view will be used during content sessions alongside Sanity Studio. It needs to be fast enough not to interrupt the editing workflow.
